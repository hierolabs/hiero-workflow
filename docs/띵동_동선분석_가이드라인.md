# 띵동(ThingDone) 청소 동선 분석 가이드라인

> 청소 선생님의 업무 시간·이동 동선·효율을 데이터로 추적하고 최적화하는 시스템

## 목표

```
AS-IS: 배정표 카톡 전달 → 선생님 자율 이동 → 결과만 확인
TO-BE: 배정 → 실시간 동선 추적 → 시간/거리 분석 → 배정 최적화
```

핵심 KPI:
- 청소 1건당 평균 소요시간 (목표: 35분)
- 건물 간 이동시간 (목표: 15분 이내)
- 일일 총 업무시간 대비 실제 청소시간 비율 (목표: 70%+)

---

## Phase 1: 단기 — 기존 데이터 기반 시간 분석 (NOW)

### 1.1 데이터 소스

현재 `cleaning_tasks` 테이블에 이미 존재하는 필드:

| 필드 | 타입 | 설명 |
|------|------|------|
| `started_at` | datetime | 청소 시작 시각 |
| `completed_at` | datetime | 청소 완료 시각 |
| `cleaner_id` | uint | 청소자 |
| `property_code` | string | 숙소 코드 (권역 추출 가능) |
| `address` | string | 숙소 주소 |
| `cleaning_date` | string | 청소 날짜 |

### 1.2 계산 가능한 지표

```
같은 청소자의 하루 업무를 시간순 정렬:

Task 1: GN201  started_at=11:00  completed_at=11:38
Task 2: GN305  started_at=11:48  completed_at=12:22
Task 3: SD102  started_at=12:40  completed_at=13:15

→ 청소시간 = completed_at - started_at
   Task 1: 38분, Task 2: 34분, Task 3: 35분

→ 이동시간 = 다음 Task의 started_at - 이전 Task의 completed_at
   1→2: 10분 (같은 권역 GN)
   2→3: 18분 (권역 이동 GN→SD)

→ 총 업무시간 = 마지막 completed_at - 첫 started_at = 135분
→ 순수 청소시간 = 38+34+35 = 107분
→ 이동시간 합계 = 10+18 = 28분
→ 효율 = 107/135 = 79%
```

### 1.3 구현 범위

**Backend API**: `GET /cleaning/time-analysis`

```
Query: ?date=2026-05-08 또는 ?start_date=...&end_date=...&cleaner_id=...

Response:
{
  "cleaners": [
    {
      "cleaner_id": 1,
      "cleaner_name": "우연",
      "date": "2026-05-08",
      "tasks": [
        {
          "order": 1,
          "property_code": "GN201",
          "address": "강남구 ...",
          "region": "GN",
          "started_at": "11:00",
          "completed_at": "11:38",
          "cleaning_minutes": 38,
          "travel_minutes_to_next": 10,
          "next_region": "GN",
          "is_cross_region": false
        },
        ...
      ],
      "summary": {
        "total_tasks": 3,
        "total_work_minutes": 135,
        "total_cleaning_minutes": 107,
        "total_travel_minutes": 28,
        "efficiency_pct": 79.3,
        "avg_cleaning_minutes": 35.7,
        "avg_travel_minutes": 14.0,
        "cross_region_moves": 1
      }
    }

}
```

**Admin UI**: 청소 관리 > 새 탭 "동선 분석"

- 날짜별 청소자 타임라인 (가로 바 차트)
- 청소시간 vs 이동시간 비율 시각화
- 권역 이동 하이라이트 (동일 권역=초록, 타 권역=빨강)
- 청소자별 평균 통계 테이블

### 1.4 한계

- `started_at`/`completed_at`를 선생님이 직접 눌러야 함 (현재 앱에서 가능)
- 누락 데이터 있을 수 있음 → "도착" 카톡으로 보정 가능
- GPS 없이 정확한 이동거리 산출 불가 → 주소 기반 직선거리로 추정

## Phase 2: 중기 — 카톡 메시지 파싱 + 보정 (1~2개월)

### 2.1 카톡 대화 패턴

청소 단톡방의 일반적 메시지 패턴:

```
[우연] [오전 11:00] 강남201 도착
[우연] [오전 11:38] 강남201 완료
[우연] [오전 11:40] 사진 3장
[우연] [오전 11:48] 강남305 도착
[우연] [오전 12:22] 강남305 완료요
[우연] [오후 12:40] 송동102 도착이요
[우연] [오후 1:15] 송동102 끝
```

### 2.2 파싱 규칙

```
메시지 분류:
- 도착: "도착", "도착이요", "도착했어요", "왔어요"
- 완료: "완료", "완료요", "끝", "끝났어요", "다했어요"
- 사진: image/photo attachment → 완료 직후면 증빙
- 이슈: "고장", "파손", "없어요", "안돼요" → 이슈 자동 감지

숙소 매칭:
- 숙소코드 직접 언급: "GN201", "B104"
- 숙소명 언급: "강남201", "송파102" → property_code 매핑
- 이전 배정표와 순서 매칭 (배정 순서대로 메시지가 오는 패턴)
```

### 2.3 구현 범위

**카톡 내보내기 파서**: `scripts/parse_kakao_cleaning.py` (또는 Go)

```
Input: 카톡 대화 내보내기 .txt 파일
Output: 시간별 이벤트 리스트

[
  {"time": "11:00", "cleaner": "우연", "property": "GN201", "event": "arrival"},
  {"time": "11:38", "cleaner": "우연", "property": "GN201", "event": "complete"},
  ...
]
```

**DB 테이블**: `cleaning_time_logs`

```sql
CREATE TABLE cleaning_time_logs (
  id            BIGINT AUTO_INCREMENT PRIMARY KEY,
  cleaning_task_id  INT,
  cleaner_id    INT,
  property_code VARCHAR(50),
  cleaning_date DATE,
  event_type    ENUM('arrival', 'start', 'complete', 'issue', 'photo'),
  event_at      DATETIME,
  source        ENUM('app', 'kakao', 'manual'),  -- 데이터 출처
  raw_message   TEXT,                              -- 원본 카톡 메시지
  created_at    DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

**보정 로직**:
- 카톡 도착 시간 → `started_at` 보정 (앱에서 안 눌렀을 때)
- 카톡 완료 시간 → `completed_at` 보정
- 앱 데이터와 카톡 데이터 교차 검증 → 정확도 리포트

### 2.4 업로드 프로세스

```
1. 관리자가 카톡 대화 내보내기 (.txt) 업로드
2. 파서가 메시지 분류 + 숙소/청소자 매칭
3. 매칭 결과 프리뷰 (확인 후 저장)
4. cleaning_time_logs에 저장
5. 기존 cleaning_tasks의 started_at/completed_at 보정 (옵션)
```

## Phase 3: 장기 — 띵동 모바일 앱 + 실시간 추적 (3~6개월)

### 3.1 띵동 모바일 앱 (별도 프로젝트)

```
프로젝트: thing-done-app (React Native 또는 PWA)
대상: 청소 선생님 전용 앱
```

**핵심 기능**:

| 기능 | 설명 |
|------|------|
| 오늘 배정 목록 | 내 청소 순서·숙소·시간 확인 |
| [도착] 버튼 | GPS 좌표 + 시간 자동 기록 |
| [완료] 버튼 | GPS + 시간 + 사진 필수 |
| 네비게이션 | 다음 숙소까지 카카오맵 길찾기 연동 |
| 이슈 신고 | 사진 + 카테고리 선택 → 즉시 관리자 알림 |
| 일일 리포트 | 오늘 내 청소시간/이동시간/효율 |

### 3.2 GPS 기반 동선 추적

```
앱에서 수집:
- 도착 GPS (위도, 경도)
- 완료 GPS
- 이동 중 경로 (선택적, 배터리 고려)

서버에서 계산:
- 실제 이동거리 (카카오 길찾기 API)
- 예상 이동시간 vs 실제 이동시간
- 최적 순서 제안 (TSP 알고리즘)
```

### 3.3 DB 확장

```sql
-- properties 테이블에 좌표 추가
ALTER TABLE properties ADD COLUMN latitude  DECIMAL(10,7);
ALTER TABLE properties ADD COLUMN longitude DECIMAL(10,7);

-- 주소 → 좌표 변환 (카카오 Geocoding API 일괄)

-- 동선 최적화 캐시
CREATE TABLE route_cache (
  id              BIGINT AUTO_INCREMENT PRIMARY KEY,
  from_property_id INT,
  to_property_id   INT,
  distance_meters  INT,       -- 실제 도로 거리
  duration_minutes INT,       -- 예상 이동시간
  transport_type   VARCHAR(20), -- walk, car, public
  updated_at       DATETIME
);
```

### 3.4 배정 최적화 엔진

현재: 권역 매칭 + 워크로드 밸런싱
개선:

```
입력:
- 오늘 청소 대상 숙소 N개 (좌표, 예상 청소시간, 체크아웃 시간)
- 청소자 M명 (시작 위치, 이동수단, 가용시간)

출력:
- 청소자별 최적 방문 순서
- 예상 완료 시각
- 총 이동거리/시간

알고리즘:
1. 권역 클러스터링 (기존)
2. 클러스터 내 TSP (Nearest Neighbor + 2-opt)
3. 체크아웃 시간 제약 반영 (빠른 체크아웃 우선)
4. 이동수단별 속도 반영 (도보 4km/h, 자전거 15km/h, 자차 25km/h)
```

## 숙소 주소 데이터 현황

현재 `properties` 테이블:

| 필드 | 설명 | 상태 |
|------|------|------|
| `region` | 권역 코드 (A, B, V 등) | O 채워짐 |
| `address` | 도로명/지번 주소 | O 채워짐 |
| `detail_address` | 상세주소 (동/호) | O 채워짐 |
| `building_name` | 건물명 | O 채워짐 |
| `latitude` | 위도 | X 미존재 |
| `longitude` | 경도 | X 미존재 |

Phase 1에서는 주소 텍스트의 권역 코드로 동일 건물/권역 판별.
Phase 3에서 카카오 Geocoding API로 좌표 일괄 변환 예정.

---

## 권역 코드 → 지역 매핑

```
A, A2      → 강동구 암사/선사 일대
B          → 강동구 천호/성내
GN         → 강남구
SD, SD2    → 송파구
V          → 기타/분산
```

같은 권역 내 이동: 도보 5~10분
권역 간 이동: 대중교통 15~30분, 자차 10~20분

## 데이터 수집 흐름 (Phase별)

```
Phase 1 (단기):
  선생님 → 앱에서 [시작]/[완료] 클릭
  → cleaning_tasks.started_at / completed_at 기록
  → time-analysis API로 분석

Phase 2 (중기):
  선생님 → 카톡방에 "도착"/"완료" 메시지
  → 관리자가 카톡 내보내기 업로드
  → 파서가 cleaning_time_logs에 저장
  → started_at/completed_at 보정
  → time-analysis API (보정 데이터 포함)

Phase 3 (장기):
  선생님 → 띵동 앱에서 [도착]/[완료] 탭
  → GPS + 시간 자동 기록
  → cleaning_time_logs (source=app)
  → 실시간 동선 지도 표시
  → 배정 최적화 엔진 피드백
```
