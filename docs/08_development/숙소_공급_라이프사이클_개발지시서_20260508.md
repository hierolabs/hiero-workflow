> 데이터 원천: 2-1 숙소 공급.xlsx + 4-1 플랫폼.xlsx + Google Sheet 온보딩 관리

## 1. Full Lifecycle

```txt
Lead → Meeting → Negotiating → Contract
→ Setting → Filming → OTA Registration → Operation Ready
→ Active
```

이건 단순 onboarding이 아니라:

```txt
공급 확보 (Lead ~ Contract)
공간 제작 (Setting ~ Filming)
디지털 배포 (OTA Registration ~ Operation Ready)
판매 운영 (Active)
```

4개 Phase야.

## 2. 상태 모델

### property_lifecycle_status

```txt
-- Phase 1: 공급 확보
lead                    — 리드 발굴
meeting                 — 미팅 진행
negotiating             — 조건 협상 중
contracted              — 계약 완료

-- Phase 2: 공간 제작
setting                 — 세팅 중 (light/standard/renovation)
filming                 — 촬영 대기/진행

-- Phase 3: 디지털 배포
ota_registering         — OTA 등록 중
operation_ready         — 운영 준비 완료

-- Phase 4: 판매 운영
active                  — 운영 중
paused                  — 일시 중단
closed                  — 폐기/해지
```

### setting_type

```txt
light           — 경정비 (3일, 1룸 기준)
standard        — 표준 세팅 (5일, 2룸 기준)
renovation      — 대수선/인테리어 (14일)
```

## 3. ETA Engine

각 단계별 예상 소요시간:

| 단계 | 예상 시간 | 변수 |
|------|---------|------|
| Lead → Contract | 7~30일 | 미팅 횟수, 지역 |
| Setting (light) | 3일 | 1룸 |
| Setting (standard) | 5일 | 2룸 |
| Setting (renovation) | 14일 | 대수선 |
| Filming | 3시간 | 세팅 완료 후 |
| OTA Registration | 6시간 | 소개글+사진+설정 |
| Operation Ready | 당일 | 체크인 안내+청소 연결 |

자동 계산:

```txt
Expected Active Date = contract_date
  + setting_days (by setting_type)
  + 1 (filming)
  + 1 (OTA)
  + 1 (operation_ready)
```

예시:
- 계약일 5/10, light 세팅 → 예상 active: 5/15
- 계약일 5/10, renovation → 예상 active: 5/27

## 4. Dependency Rules

### Hard Dependency (강제 순차)

```txt
Contract 전 Setting → BLOCK
Setting 전 Filming → BLOCK
사진 없이 OTA 등록 → BLOCK (최소 사진 필수)
OTA 미등록 Active → BLOCK
```

### Soft Dependency (병렬 가능)

```txt
Filming 전 소개글 작성 → 가능
Setting 중 가격 정책 준비 → 가능
OTA 일부 플랫폼만 등록 → 가능 (나머지 pending)
```

## 5. 플랫폼별 등록 상태

### property_platforms 테이블

```sql
property_platforms (
  id,
  property_id,          -- properties FK
  platform,             -- hostex, airbnb, booking, agoda, 33m2, liv, jaritalk, naver
  platform_name,        -- 플랫폼 내 숙소명
  platform_url,         -- 플랫폼 URL
  status,               -- draft, pending, review, active, paused, rejected
  host_name,            -- 담당 호스트
  registered_at,
  activated_at,
  memo,
  created_at,
  updated_at
)
```

### 상태값

```txt
draft       — 초안 작성 중
pending     — 등록 제출, 승인 대기
review      — 플랫폼 검토 중
active      — 운영 중
paused      — 일시 중지
rejected    — 거절/반려
```

## 6. 온보딩 체크리스트

### onboarding_checks 테이블

```sql
onboarding_checks (
  id,
  property_id,
  phase,                -- 1(setting), 2(hiero_register), 3(platform), 4(listing), 5(operation)
  item,                 -- 체크 항목명
  is_checked,           -- bool
  checked_by_id,        -- 체크한 사람
  checked_by_name,
  checked_at,
  memo,
  created_at
)
```

### Phase별 항목 (엑셀 기반)

```txt
Phase 1. 공간 세팅 (진태)
  □ 공간 컨셉 확정
  □ 물품 구매 완료
  □ 설치 완료
  □ 도어락 설정
  □ 와이파이 설정
  □ 어메니티/비품 배치
  □ 침대/침구 세팅 (등급별: S/D/P)

Phase 2. 촬영 + 디지털 (진태 → 예린)
  □ 사진 촬영 완료
  □ 색보정 완료
  □ AI 수정 완료
  □ 사진 업로드 완료

Phase 3. 콘텐츠 (예린)
  □ 소개글 작성
  □ 소개글 가이드 (교통편, 편의시설)
  □ 위치 정보 확정
  □ 인천공항 ↔ 숙소 교통 안내
  □ 주변 편의시설 안내

Phase 4. 플랫폼 등록 (예린)
  □ Hostex 등록
  □ Airbnb 등록
  □ Booking.com 등록
  □ Agoda 등록
  □ 삼삼엠투 등록
  □ 리브애니웨어 등록
  □ 자리톡 등록
  □ 가격 설정 (플랫폼별)
  □ 최소 숙박일 설정

Phase 5. 운영 준비 (재관)
  □ 체크인 안내 메시지 준비
  □ 하우스룰 설정
  □ 청소 스케줄 등록
  □ Hostex 연동 확인
  □ 첫 예약 수신 확인
```

## 10. Founder Dashboard 연결

김진우가 봐야 하는 것:

```txt
공급 파이프라인 요약:
┌─────────────────────────────────────────┐
│ Lead: 5  → Meeting: 3  → 계약대기: 2    │
│ 세팅중: 4 → 촬영대기: 1 → OTA등록: 2    │
│ 준비완료: 1 → Active: 76               │
│                                        │
│ 예상 신규 Active: 이번달 3개 (5/15, 5/20, 5/27)│
│ 병목: 촬영 지연 1건, OTA 지연 2건         │
└─────────────────────────────────────────┘
```

### KPI 추가

| KPI | 목표 | 측정 |
|-----|------|------|
| Lead → Contract 전환율 | ≥ 30% | contracted / total_leads |
| 평균 온보딩 기간 | ≤ 10일 | contract → active 평균 |
| 촬영 지연율 | ≤ 10% | filming 48h 초과 비율 |
| OTA 등록 완료율 | ≥ 80% | 4개+ 플랫폼 등록 비율 |
| Active Readiness Score | ≥ 90% | 체크리스트 완료율 |

## 11. 병목 감지 (Bottleneck Detection)

자동 이슈 생성 조건:

```txt
세팅 7일 초과 (light 기준) → P1 이슈 → 진태 배정
촬영 대기 48시간 초과 → P1 이슈 → 진태 배정
OTA 등록 3일 초과 → P1 이슈 → 예린 배정
Active 예정일 초과 → P0 이슈 → CEO 에스컬레이트
```
