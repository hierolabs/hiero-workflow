# HIERO 가격 자동화 운영매뉴얼

> **버전**: v0.1  
> **최종 수정일**: 2026-05-08  
> **대상 독자**: HIERO 운영팀, 개발팀  
> **목적**: PriceLabs–Hostex–Airbnb 흐름 위에 HIERO가 얹히는 자동 가격 시스템의 운영 기준을 문서화

---

## 0. 한눈에 보기

```
[PriceLabs]              [Hostex]              [Airbnb/OTA]
시장분석+AI 추천가  →    채널 허브        →     판매
                              ↑
                          [HIERO]
                       관찰자 + 의사결정자
                       (DB 허브 + 온톨로지)
```

**HIERO는 가격을 "결정"하지 않는다. HIERO는 가격이 어떻게 흐르는지 보고, 이상하면 알리고, 필요하면 사람이 개입할 수 있게 한다.**

## 1. 시스템 아키텍처

### 1.1 가격이 흐르는 경로

1. **PriceLabs** — 시장 데이터 + AI로 추천가 생성
2. **Hostex** — 채널 허브. 추천가/수동가를 받아 OTA에 동기화
3. **Airbnb 등 OTA** — 실제 판매 채널
4. **HIERO** — 1·2·3을 모두 캐시·기록·관찰. 가격 결정 자체는 PriceLabs와 사람의 몫

### 1.2 역할 분담

| 구분 | 주체 | HIERO가 하는 일 |
|---|---|---|
| 가격 결정 | PriceLabs | 추천가 vs 현재가 비교, 이상치 알림 |
| 채널 동기화 | Hostex | DB에 캐시, 변경 이력 추적 |
| 판매 | Airbnb | 예약 데이터 수집, 매출 분석 |
| 수동 개입 | HIERO | 특정 날짜 가격 직접 변경 (Hostex 경유) |

### 1.3 현재 구현 상태

**구현 완료**
- `listing_calendars` DB 캐시
- 예약 웹훅 → `reservations` 테이블
- PriceLabs 데이터 수집 → `price_labs_prices` 테이블
- 수동 가격 편집 → Hostex 푸시

**미구현 (TODO)**
- PriceLabs 추천가 vs 현재가 자동 알림 (이상치 탐지)
- 가격 변경 이력 추적 (누가, 언제, 왜)
- 기본가/주말가 규칙 기반 자동 설정

## 2. 1단계: 분석 패키지 (먼저 만들 것)

자동 가격 결정의 근거가 되는 분석 레이어. **자동화는 분석 결과 위에 얹는다.**

### 2.1 분석 축

`숙소별 x 요일별 x 채널별` 가격/가동률/ADR 매트릭스를 기준으로 한다.

### 2.2 데이터 소스 매핑

| 분석 축 | 데이터 소스 | 산출물 |
|---|---|---|
| 숙소별 | `listing_calendars` + `reservations` | 숙소별 평균가, 가동률, ADR |
| 채널별 | `reservations.channel_type` | 비엔비 vs 삼삼엠투 vs 리브 단가 차이 |
| 요일별 | `check_in_date` → 월~일 변환 | 요일별 가동률/평균가 히트맵 |
| 기간별 | `hostex_transactions` | 월별 추이, 시즌 패턴 |

### 2.3 핵심 산출물

- **가격 기준선** (숙소별 x 요일별 적정가)
- 인사이트형 알림 예시: *"이 숙소는 화요일에 항상 비엔비에서 안 팔린다"*

## 3. 2단계: 공실 기반 자동 가격 엔진

### 3.1 매일 자동 실행 (cron)

```
1. 오늘 ~ 14일 후 공실 수 계산
   공실 = 전체 active 숙소 - 예약된 숙소 - 차단된 숙소

2. 공실 등급 판정 (아래 표)

3. 등급별 기준가 조정 → Hostex API 푸시

4. 미판매 지속 시 5시간 타이머 시작
```

### 3.2 공실 등급 판정 기준

| 14일 내 공실 수 | 등급 | 기준가 조정 |
|---|---|---|
| 30개 이상 | **EMERGENCY** | -25% |
| 15 ~ 29개 | **LOW** | -10% |
| 10 ~ 14개 | **NORMAL** | 유지 |
| 9개 이하 | **HIGH** | +15% |

### 3.3 5시간 타이머 (단계적 하락)

기준가 조정 후에도 안 팔리면:

- 첫 5시간 후 미판매 → 추가 **-5%**
- 다시 5시간 후 → 추가 **-5%**
- **최저가 하한 도달 시 멈춤** (4.1 참조)

## 4. 핵심 안전장치

자동화가 잘못되면 손해가 직접적으로 발생한다. 아래 4개는 **타협 불가**.

### 4.1 최저가 하한 (Floor)

- 숙소별 **손익분기점** 이하로는 **절대 안 내림**
- 숙소 등록 시 손익분기 가격을 명시적으로 설정해두어야 함
- DB에 `min_price` 컬럼으로 보관

### 4.2 수동 오버라이드

- 모든 자동 변경은 사람이 **즉시 되돌릴 수 있어야 함**
- Admin UI에 "되돌리기" 버튼 + 잠금(lock) 기능
- 잠금된 가격은 자동 엔진이 건드리지 않음

### 4.3 로그 (Audit Trail)

모든 자동 변경 이력을 기록한다.

| 필드 | 내용 |
|---|---|
| `when` | 변경 시각 |
| `why` | 트리거 (공실등급/타이머/수동) |
| `what` | 이전가 → 현재가 |
| `who` | 자동 / 어떤 admin user |

### 4.4 시작은 알림만 (Dry Run)

처음 배포 시에는 자동 변경 대신 **"이 가격으로 바꿔야 합니다" 알림만** 발송 → 1~2주 검증 → 검증 후 자동화 ON.

## 5. 수요 선행지표 (Phase 2)

> **핵심: PriceLabs보다 먼저 수요를 감지하면 가격 선점이 가능하다.**

### 5.1 선행 지표 후보

| 지표 | 데이터 소스 | 리드타임 | 실현성 |
|---|---|---|---|
| KSPO 콘서트 일정 | KOPIS API | 1~3개월 전 | 높음 (공공/무료) |
| 주요 행사/축제 | 한국관광공사 Tour API | 1~2개월 전 | 높음 (공공) |
| 외국인 입국 통계 | 출입국관리사무소 공공데이터 | 후행(월별) | 패턴 학습용 |
| 항공편 예약 추이 | OAG/Skyscanner API | 2~4주 전 | 중간 (유료) |
| **자체 예약 속도** | `reservations.booked_at` | 실시간 | **높음 (이미 있음)** |
| 동남아 명절 캘린더 | 정적 데이터 | 확정 | 높음 (직접 입력) |

### 5.2 가장 현실적인 조합

**[즉시 가능] 자체 예약 속도 감지**
```
"이 날짜에 평소 대비 3배 빨리 예약이 찬다"
→ HIERO 알림: "5/17~18 KSPO BTS 콘서트
              현재 가격 110,000원
              추천 180,000원+
              PriceLabs base_price 올려야 함"
```

**[1~3개월 선행] KOPIS API 연동**
- 공연예술통합전산망(KOPIS) — 문화체육관광부 운영, 무료
- KSPO 돔, 올림픽공원, 잠실 등 **장소 코드로 필터링**
- 공연명/날짜/장르/예매 현황 제공
- 활용 흐름:
  1. KSPO 주변 숙소 자동 태깅
  2. 콘서트 날짜 → 해당 숙소 가격 알림
  3. *"이 날 PriceLabs base_price를 X만 원으로 올리세요"* 제안

**[정적 캘린더] 동남아 명절**
- 태국 송크란 (4/13~15)
- 중국 노동절 (5/1~5)
- 일본 골든위크 (4/29~5/5)
- 말레이시아 하리라야 (매년 변동)
- 인도네시아 르바란 (매년 변동)
- → DB에 넣고 해당 시기 **2주 전부터** 가격 제안 알림

## 6. Demand-Aware Markdown Engine

### 6.1 운영 철학

오재관 팀장: "미래 가격은 높게 열고, 안 팔리면 시간 지나면서 내린다"
→ 철학은 맞지만, HIERO에서는 감이 아니라 데이터 기반으로.

### 6.2 5대 변수 카테고리

#### 1. 내부 공실 변수
- vacancy_count, vacancy_rate, days_to_checkin, unsold_hours
- `urgency = vacancy_rate x lead_time_weight`
- D-30 공실 30개 = 정상 / D-3 공실 30개 = 비상 / D-1 공실 10개 = 비상

#### 2. 시장 공급량 변수
- market_available_count/rate, competitor_median/lowest_price, competitor_price_rank
- 우리 40만원인데 주변 중간가 22만원 → 노출/전환 손상
- 주변 다 팔림 → 유지/상향 가능

#### 3. 채널별 성과 변수
- channel_booking_pickup, conversion, net_revenue, price_sensitivity
- Airbnb pickup 급락 + 삼삼엠투 유지 = Airbnb 가격/노출 문제
- 전체 예약만 보면 안 됨, 채널별로 봐야

#### 4. 이벤트/명절 수요 변수
- event_score, holiday_score, foreign_inbound_score, flight_score
- event_score 높음 + market_supply 낮음 + pickup 빠름 = 가격 상승
- event_score 높음 + market_supply 많음 + pickup 느림 = 유지 또는 소폭 하락

#### 5. 예약 속도 변수
- pickup_velocity, expected_pickup, pickup_gap
- "안 팔렸냐"가 아니라 "예상보다 느리냐"
- D-20 기준 40% 팔려야 하는데 10%만 → 가격 위험

### 6.3 가격 함수

```
target_price =
  base_price
  x event_multiplier
  x market_supply_multiplier
  x vacancy_pressure_multiplier
  x channel_multiplier
  x lead_time_multiplier
```

### 6.4 5시간 미판매 하락

```
if unsold_hours >= 5:
  price = current_price x (1 - markdown_rate)

markdown_rate =
  기본하락률 (3%)
  + 공실압박 (+0~4%)
  + 시장공급압박 (+0~3%)
  - 이벤트수요보정 (-0~2%)
```

### 6.5 "안 팔리는 이유" 6가지 분석

| # | 이유 | 데이터 소스 | 확보 상태 |
|---|------|-----------|----------|
| 1 | 적정 예약시간/빈도 | Airbnb booking pickup | PriceLabs booking_pickup_30 |
| 2 | 시장 공급 과다 | PriceLabs 주변 분석 | market_occupancy 있음, 공급량 직접 제공 여부 확인 필요 |
| 3 | 가격이 경쟁 대비 높음 | PriceLabs mpi_next_7 | 있음 (1.0 이상 = 시장보다 비쌈) |
| 4 | 채널 노출 약화 | Airbnb 주력, 현재 강함 | 자체 예약 데이터로 간접 추적 |
| 5 | 이벤트 수요 대기 | KOPIS/TourAPI/동남아명절 | 연동 예정 |
| 6 | 전년 동기 패턴 | 자체 1년 예약 데이터 | reservations 테이블에 있음 |

## 7. 외부 신호 API

### 7.1 분류

```
수요 예측 API
- KOPIS: 공연/콘서트
- TourAPI: 축제/관광 이벤트
- 항공/입국: 외국인 관광 수요

운영 리스크 API
- 기상청: 가격 + 청소 + 이슈 + CS
```

### 7.2 기상청 = 전사 운영 리스크 신호

> HIERO 외부 신호 연동에서 기상청 API는 단순 가격 수요 변수가 아니라 **전사 운영 리스크 신호**로 사용한다. 기상 악화는 가격, 청소 배정, 시설 점검, 게스트 CS 안내에 모두 영향을 주는 **공통 컨텍스트**로 설계한다.

| 영역 | 기상 영향 |
|------|----------|
| 가격 | 비 오는 주말 예약률 저하, 폭설/태풍 단기 예약 하락 |
| 청소 | 폭우/폭설 이동 시간 가중, 세탁/건조 지연 |
| 시설 이슈 | 한파(보일러/동파), 폭우(누수/습기), 폭설(출입로), 폭염(에어컨) |
| 게스트 CS | 체크인 안내 강화, 교통/난방 안내, 취소/변경 증가 |

### 7.3 API 발급 상태

| API | 사이트 | 상태 |
|-----|--------|------|
| KOPIS | kopis.or.kr | 미발급 |
| TourAPI | api.visitkorea.or.kr | 미발급 |
| 항공운항 | data.go.kr | 미발급 |
| 기상청 | data.go.kr | 미발급 |

```sql

  id, listing_id, changed_at,

demand_signals (
  id, signal_date, signal_type,  -- 'concert' | 'holiday' | 'festival'
  source,                         -- 'kopis' | 'static' | 'manual'
  description, affected_area,
  expected_impact                 -- 'high' | 'medium' | 'low'
)
```

## 11. FAQ / 운영 시 주의사항

**Q. PriceLabs 추천가와 우리 자동 엔진 가격이 충돌하면?**
A. 우선순위는 `수동 락 > HIERO 자동 엔진 > PriceLabs`. PriceLabs는 base_price 추천만 받고, 실시간 조정은 HIERO 엔진이 한다.

**Q. 5시간 타이머가 도는 동안 예약이 들어오면?**
A. 즉시 타이머 종료. 다음 cron에서 재평가.

**Q. 최저가 하한을 일시적으로 풀어야 할 때 (예: 폭우, 비수기)?**
A. Admin UI에서 숙소별 `min_price`를 수정. 변경 이력은 로그에 남는다.

**Q. 한 숙소에서 자동 변경이 너무 자주 일어나면?**
A. `is_locked`를 켜고 수동 모드로 전환. 원인 분석 후 다시 자동 모드로.
