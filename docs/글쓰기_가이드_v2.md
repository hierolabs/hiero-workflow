# HIERO 글쓰기 가이드 v2

> 코드는 기록이고, 글은 자산이다.
> 하나의 작업이 8개의 탭으로 쌓이고, 책이 된다.

---

## 전체 구조 — 8탭 파이프라인

하나의 위키 아티클 = 8개 탭. content에 `<!-- TAB: 이름 -->` 구분자를 넣으면 자동 탭 UI 생성.

```
[내부 기술 문서 — 기록]
  TAB 1. 작업 기록        ← 매 세션, Claude 자동 (draft)
  TAB 2. 시스템 흐름도    ← 작업 기록 기반, Claude 작성 (draft)
  TAB 3. 개념 설명        ← 작업 기록 기반, Claude 작성 (draft)
  TAB 4. 업무 지침        ← 운영팀 관점, Claude 작성 (draft)

[외부 콘텐츠 — 글]
  TAB 5. 에세이           ← 주 1회, Gemini 초안 → Claude 검수 (review)
  TAB 6. 논문형           ← 주 1회, Gemini 초안 → Claude 검수 (review)
  TAB 7. 블로그           ← 주 1회, Gemini 초안 → Claude 검수 (review)
  TAB 8. 검토/퇴고        ← 월 1회, Founder + CTO 직접 (published)
```

### 상태 매핑

| 위키 status | 탭 완성 범위 | 설명 |
|---|---|---|
| empty | — | 아무것도 없음 |
| draft | TAB 1~4 진행 중 | 내부 기술 문서 작성 단계 |
| review | TAB 5~7 완성 | 3종 초안 작성됨, 사람 검토 대기 |
| published | TAB 8 완료 | Founder + CTO 퇴고 후 확정 |

### 주기

```
매 세션 종료:  TAB 1 자동 저장 → TAB 2~4 함께 작성 (draft)
주 1회 금요일: TAB 1~4 모아서 → TAB 5~7 초안 생성 (review)
월 1회:        TAB 8 퇴고 → published → 외부 발행
```

## TAB 1. 작업 기록

> **목적**: 모든 개발 작업의 원본 소스. 6개월 후의 내가 읽어도 "왜 이걸 만들었는지" 알 수 있어야 한다.
> **작성자**: Claude (매 세션 종료 시 자동)
> **대상 독자**: 기술자, 미래의 나, 다음 세션의 Claude

### 기록 원칙

- **왜** 이 작업이 필요했는가 (배경/맥락)
- **어떤 구조**로 만들었는가 (설계 흐름)
- **무엇이** 바뀌었는가 (코드/데이터 변경)
- **어떤 결정**을 했는가 (왜 A를 선택하고 B를 버렸는가)
- **다음엔** 뭘 해야 하는가 (이어질 작업)

### 작성 템플릿

```markdown
<!-- TAB: 작업 기록 -->

## [작업 제목]
**날짜**: 2026-MM-DD
**세션**: #N

### 배경
왜 이 작업이 필요했는가.
어떤 문제를 풀려고 했는가.
이전 세션에서 어디까지 했는가.

### 구조
어떻게 설계했는가.
흐름도, 테이블 구조, API 설계.

### 변경 내역
| 파일 | 변경 내용 |
|------|----------|
| backend/handler/xxx.go | 신규 — OOO 핸들러 |
| backend/service/xxx.go | 수정 — OOO 로직 추가 |
| admin/src/pages/Xxx.tsx | 수정 — OOO 컴포넌트 |

### 기술 결정
| 결정 | 선택 | 대안 | 이유 |
|------|------|------|------|
| OOO | A 방식 | B 방식 | ~때문에 |

### 데이터
- 테이블: xxx (N건 추가/변경)
- 소스: API / CSV / 수동입력
- 검증: OOO 확인함

### 다음 단계
- [ ] 이어서 해야 할 것 1
- [ ] 이어서 해야 할 것 2
```

## TAB 2. 시스템 흐름도

> **목적**: 이 기능이 전체 시스템에서 어떻게 연결되는지 보여준다. 데이터가 어디서 시작해서 어디로 흐르는지. 의사결정 포인트는 어디인지. 코드는 어디에 있는지.
> **작성자**: Claude (작업 기록 기반)
> **대상 독자**: 기술자, 신규 팀원, 투자자(기술 이해용)

### 작성 원칙

- **입력 → 처리 → 출력** 흐름이 한눈에 보여야 한다
- **의사결정 포인트**를 명시한다 (자동/수동/에스컬레이션)
- **코드 위치**를 함께 적는다 (파일:함수명)
- **관련 엔티티**를 표시한다 (어떤 테이블이 관여하는가)
- 텍스트 기반 다이어그램 사용 (```로 감싸기)

### 작성 템플릿

```markdown
<!-- TAB: 시스템 흐름도 -->

## 데이터 흐름

```
[트리거] ──→ [처리 1] ──→ [처리 2] ──→ [결과]
                │                          │
                ▼                          ▼
          [의사결정]                  [저장/알림]
          자동 or 수동
```

## 관련 코드

| 단계 | 파일 | 함수/핸들러 |
|------|------|------------|
| 트리거 | backend/handler/xxx.go | HandleXxx() |
| 처리 | backend/service/xxx.go | ProcessXxx() |
| 저장 | backend/models/xxx.go | XxxModel |

## 관련 엔티티

| 엔티티 | 역할 | 상태 전환 |
|--------|------|----------|
| Reservation | 입력 | created → confirmed |
| CleaningTask | 출력 | pending → assigned |

## 의사결정 포인트

| 포인트 | 조건 | 자동/수동 | 에스컬레이션 |
|--------|------|----------|-------------|
| 배정 대상 | 권역 매칭 | 자동 제안 | 미매칭 시 관리자 |
```

## TAB 3. 개념 설명

> **목적**: "왜 이렇게 만들었는가"를 비기술자도 이해할 수 있게 설명한다. 코드가 아니라 **생각**을 기록한다. 도시계획적 관점에서의 해석을 포함한다.
> **작성자**: Claude (작업 기록 기반)
> **대상 독자**: CEO, 투자자, 도시계획 연구자, 비기술 팀원

### 작성 원칙

- **비유**를 쓴다 — 도시계획 ↔ HIERO 매핑
- **선택의 이유**를 설명한다 — "이것도 됐지만 저것을 선택했다"
- **한계를 인정**한다 — "이건 아직 안 된다, 왜냐하면"
- **도시계획 부문과 연결**한다 — 이 기능이 토지이용/인구/교통/환경/경제/안전 중 어디에 해당하는가
- 코드, API, 테이블명 **금지** — 개념만

### 작성 템플릿

```markdown
<!-- TAB: 개념 설명 -->

## 한 줄 정의
이 기능은 OOO이다. 도시계획에서 OOO에 해당한다.

## 왜 이렇게 만들었는가
- 문제: ...
- 기존 방식: ...
- 우리 방식: ...
- 차이: ...

## 도시계획적 해석
도시관리계획의 [부문] 관점에서 보면, 이것은 [비유]와 같다.
예: "reservation_code는 주민등록번호다. 한 사람의 모든 행정 기록을 연결하는 키."
예: "청소 배정은 환경 관리다. 도시의 쓰레기 수거 노선 최적화와 같은 문제."

## 선택하지 않은 것
- 대안 A: ... → 왜 안 했는가
- 대안 B: ... → 왜 안 했는가

## 한계와 다음
- 지금 안 되는 것: ...
- 다음에 풀어야 할 것: ...
```

## TAB 4. 업무 지침

> **목적**: 이 기능을 실무에서 쓰는 사람을 위한 가이드. "어떻게 쓰는가"가 아니라 "무슨 상황에서, 왜, 어떤 판단 기준으로 쓰는가".
> **작성자**: Claude (작업 기록 기반)
> **대상 독자**: 운영팀 (오재관, 우연, 김진태, 박수빈)

### 작성 원칙

- **3모드**로 구분한다 — 의사결정 / 관리 / 실행
- **직급명 금지** — 모드명만 사용
- **중복 금지** — 공통 데이터 설명은 따로, 모드별 탭은 액션만
- **트리거 조건**을 명시한다 — "이 상황이 되면 이걸 한다"
- **금지 사항**을 명시한다 — "이건 절대 하지 마라"

### 작성 템플릿

```markdown
<!-- TAB: 업무 지침 -->

## 이 기능은 언제 쓰는가
- 트리거: OOO 상황이 발생하면
- 주기: 매일 / 주간 / 이벤트 발생 시

---

### 의사결정 모드
> "이걸 바꿔야 하나?"

**확인할 것:**
- [ ] 지표 A가 기준 이하인가?
- [ ] 지난주 대비 변화가 있는가?

**판단 기준:**
- OOO이면 → 조치 A
- OOO이면 → 에스컬레이션

---

### 관리 모드
> "이상 없나?"

**일일 루틴:**
1. 아침 09:00 — OOO 확인
2. 오후 18:00 — OOO 확인

**에스컬레이션 기준:**
- 24시간 미처리 → 상위 보고
- 금액 50만원 이상 → Founder 직접

---

### 실행 모드
> "지금 뭘 하지?"

**할 일:**
1. OOO 클릭
2. OOO 확인
3. OOO 처리

**금지 사항:**
- ❌ OOO 하지 마라
- ❌ OOO 없이 진행하지 마라
```

## TAB 6. 논문형

**톤**: 정제되고 구조적. 주장 → 근거 → 결론.
**분량**: 1,500~3,000자

**필수 구조:**

```

5. 결과 — 수치로 뭐가 바뀌었는가
```
