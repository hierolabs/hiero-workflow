# CTO 아카이빙 프로세스

> **목적**: 개발 과정 자체를 콘텐츠 자산으로 축적하여 블로그 → 백서 → 강의자료 → 책으로 이어지는 파이프라인 구축
> **최종 수정**: 2026-05-08

---

## 1. 아카이빙이 왜 중요한가

```
코딩은 누구나 한다.
만드는 과정을 자산으로 남기는 것이 CTO의 핵심 기술이다.

코드 → 블로그 포스트
설계 결정 → 에세이 / 백서
데이터 분석 → 논문 / 리포트
전체 과정 → 책 / 강의
```

HIERO 개발 과정 = 서울 AI 도시계획 연구소(Odyssey-X)의 교육 콘텐츠 원천

---

## 2. 기록 체계 — 3계층

### 계층 1: 실시간 기록 (매 세션)

| 기록물 | 저장 위치 | 자동/수동 |
|--------|----------|----------|
| 코드 변경 | Git 커밋 | 수동 (세션 종료 시) |
| 기술 결정 메모 | Claude memory (handoff_*.md) | 수동 |
| 설계 문서 | docs/ 폴더 | 수동 |
| 데이터 분석 결과 | docs/ 또는 DB | 반자동 |

### 계층 2: 구조화 기록 (Hestory 위키)

| Part | 내용 | 담당 |
|------|------|------|
| Part 0 | HIERO Ontology | CTO |
| Part 1 | Vision & Structure + BM | CEO/CTO |
| Part 2 | HIERO Core (예약/청소/이슈) | Operations/CTO |
| Part 3 | ThingDone (청소 SaaS) | Cleaning/CTO/CFO |
| Part 4 | Back Office (정산/비용) | CFO/CTO |
| Part 5 | AI Agent 시스템 | CTO/CFO |
| Part 6 | Scale-up 전략 | CEO/CTO |
| Part 7 | Business Model | CEO |
| Part 8 | ETF 보고서 | CEO/CFO/CTO |
| Part 9 | 공문서·계약 | 전원 |
| Part 10 | 운영 데이터 자동 축적 | CTO |
| 부록 | 파일맵/API/DB/계정과목/세무 | CTO |

### 계층 3: 콘텐츠 변환 (외부 발행)

```
Hestory 위키 (내부 축적)
    │
    ├─ 블로그 포스트 (기술 결정 1개 = 1포스트)
    │   예: "92개 숙소의 가격을 어떻게 결정하는가"
    │   예: "삼삼엠투 크롤링으로 경쟁 분석 자동화하기"
    │
    ├─ 백서 (모듈 1개 = 1백서)
    │   예: "HIERO 가격 자동화 백서 v1"
    │   예: "숙박 운영 OS의 데이터 아키텍처"
    │
    ├─ 강의자료 (Part 1개 = 강의 1회분)
    │   예: "Odyssey-X 1기: 숙박 운영 디지털화"
    │
    └─ 책 (전체 = 1권)
        예: "도시공학자가 만든 숙박 운영 OS"
```

## 3. 세션별 아카이빙 흐름

```
세션 시작
    │
    ├─ 이전 핸드오프 확인
    ├─ 작업 진행
    │   ├─ 코드 변경 → Git
    │   ├─ 설계 결정 → docs/ + memory/
    │   └─ 데이터 분석 → docs/ + DB
    │
    ▼
세션 종료 전 필수 5단계
    │
    ├─ 1. Git 커밋 (코드 + docs 변경)
    ├─ 2. 위키 저장 (Hestory 해당 섹션 업데이트)
    ├─ 3. 핸드오프 메모리 (handoff_session_*.md)
    ├─ 4. docs/ 아카이빙 확인
    └─ 5. 요약 보고
```

### 기술 결정 기록 (블로그 후보)
1. **"HIERO는 가격을 결정하지 않는다"** — PriceLabs 위의 판단 레이어로 포지셔닝
2. **"70%는 사람이 결정한다"** — 삼삼엠투/리브 채널의 수동 가격 결정을 데이터로 지원
3. **"기상청 API = 가격이 아니라 운영 리스크"** — 외부 신호를 모듈별이 아니라 전사 컨텍스트로 설계
4. **"게시가 vs 체결가 +30~67% 차이"** — 관리비/청소비 누락이 가격 분석을 왜곡하는 문제
5. **"단기 +11%, 장기 -18%"** — 기간별 할인 구조의 데이터 기반 도출

## 5. 위키에 올릴 때 작성 가이드

### 기술 결정 기록 템플릿
```markdown
## [결정 제목]
- **날짜**: 2026-05-08
- **맥락**: 왜 이 결정이 필요했는가
- **선택지**: 어떤 옵션이 있었는가
- **결정**: 무엇을 선택했는가
- **근거**: 왜 그 선택인가 (데이터, 제약, 철학)
- **결과**: 이후 어떻게 됐는가 (나중에 업데이트)
```

### 데이터 분석 기록 템플릿
```markdown
## [분석 제목]
- **날짜**: 2026-05-08
- **데이터**: 어디서 온 데이터인가 (테이블, 건수)
- **방법**: 어떻게 분석했는가
- **발견**: 핵심 인사이트 3줄
- **액션**: 이 발견으로 뭘 바꿨는가
- **파일**: docs/에 저장된 위치
```

## 6. 블로그 → 백서 → 책 변환 규칙

| 원본 | 변환 | 조건 |
|------|------|------|
| 기술 결정 1개 | 블로그 1편 | 다른 사람이 읽어도 배울 수 있는 내용 |
| 분석 결과 1개 | 블로그 1편 | 구체적 숫자 + 인사이트가 있을 때 |
| 모듈 완성 | 백서 1편 | 기획→설계→구현→결과까지 완결된 모듈 |
| Part 1개 | 강의 1회 | Odyssey-X 커리큘럼에 매핑 |
| 전체 | 책 1권 | 모든 Part가 published 상태 |

## 7. 자동화 목표 (향후)

```
1단계 (현재): 수동 기록 + Claude가 종료 시 체크리스트 제안
2단계: 커밋 메시지 → 자동 블로그 초안 생성 (AI 파이프라인)
3단계: Hestory 위키 → 자동 백서 PDF 생성
4단계: 전체 축적 → 책 원고 자동 구조화
```
