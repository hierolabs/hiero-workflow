# HIERO 글쓰기 가이드 v1

> 코드는 기록이고, 글은 자산이다.
> 같은 작업이 6가지 형태의 콘텐츠가 된다.

---

## 전체 흐름

```
작업 기록 (자동)
    │
    ├─ 코드/설계/데이터가 왜, 어떻게 만들어졌는가
    │
    ▼
글 구성 (AI)
    │
    ├─ Gemini가 초안 작성 → Claude가 검수/수정
    │
    ▼
초안 3종
    │
    ├─ 에세이: 내가 느끼고 배운 것
    ├─ 논문형: 논리적 인과관계 + 근거
    └─ 블로그: 대외 공감대 형성
    │
    ▼
검토/퇴고
    │
    ├─ Founder + CTO가 직접 다듬어서 컬럼 완성
    │
    ▼
발행 → 외부 발행 (블로그/백서/강의/책)
```

---

## 1단계: 작업 기록

> 모든 개발 작업의 원본 소스. 기술자가 보고 이해할 수 있어야 한다.

### 기록 원칙
- **왜** 이 기술이 구현되었는가 (배경/맥락)
- **어떤 구조**로 만들어졌는가 (설계 흐름)
- **무엇이** 바뀌었는가 (코드/데이터 변경)
- **다음엔** 뭘 해야 하는가 (이어질 작업)

### 작성 템플릿

```markdown
## [작업 제목]

### 배경
왜 이 작업이 필요했는가. 어떤 문제를 풀려고 했는가.

### 구조
어떻게 설계했는가. 흐름도, 테이블 구조, API 설계.

### 변경 내역
- 파일: 무엇을 바꿨는가
- 데이터: 무엇이 추가/변경됐는가
- 결정: 왜 A를 선택하고 B를 버렸는가

### 다음 단계
이 작업 이후 뭘 해야 하는가.
```

---

## 2단계: 글 구성 (AI 협업)

> 작업 기록(소스)을 사람이 읽을 수 있는 글로 변환한다.

### 프로세스

```
1. Gemini에게 작업 기록을 주고 3종 초안 요청
2. Claude가 각 초안을 검수/수정
3. 위키에 저장
```

### Gemini 프롬프트 (글 구성 요청)

```
아래는 HIERO 프로젝트의 개발 작업 기록이다.
이 기록을 바탕으로 3가지 글을 작성해줘.

[작업 기록 붙여넣기]

---

1번 글: 에세이 (자유 형식)
- 이 작업을 하면서 느낀 점, 배운 것, 앞으로 배울 것
- 1인칭 시점, 솔직한 톤
- 도시공학 전공자가 기술 사업을 만드는 과정의 이야기
- 분량: 800~1,500자

2번 글: 논문형 (구조적 형식)
- 배경 → 문제 정의 → 접근 방법 → 구현 → 결과 → 시사점
- 인과관계가 명확해야 함
- 근거 데이터, 수치, 비교 포함
- 참고할 만한 외부 자료나 유사 사례 출처를 함께 제시
- 분량: 1,500~3,000자

3번 글: 블로그 (대외 소통용)
- 일반인이 이해할 수 있는 쉬운 언어
- "우리는 이런 문제를 이렇게 풀고 있다"는 메시지
- 공감대 형성이 목적 — 왜 이게 중요한지, 왜 우리가 하는지
- 도시공학 + 기술 + 숙박 운영의 교차점을 보여줘
- 분량: 1,000~2,000자
```

### Claude 검수 프롬프트

```
아래 3개의 글 초안을 검수해줘.

[3개 초안 붙여넣기]

검수 기준:
- 에세이: 진정성이 느껴지는가? 너무 AI스럽지 않은가?
- 논문형: 논리 구조가 탄탄한가? 근거가 빠진 부분은?
- 블로그: 비전문가도 이해할 수 있는가? 핵심 메시지가 반복되는가?

각 글에 대해:
1. 수정한 부분과 이유
2. 보완이 필요한 부분 표시
3. 수정된 글 전문
```

## 3단계: 초안 3종

### 3-1. 에세이

> 내가 느끼고, 생각하고, 배운 것을 자유롭게 남기는 글

**목적**: 생각 정리, 학습 기록, 성장 아카이빙
**대상**: 나 자신, 팀, 미래의 나
**톤**: 솔직하고 자유롭게. 격식 없음. 의문도, 실패도, 감탄도 다 OK.

**이런 글이 나와야 한다:**

```
가격을 자동으로 결정하겠다는 생각이 처음엔 당연했다.
PriceLabs가 추천하고, Hostex가 밀어넣고, 에어비앤비에서 팔리면 끝.

근데 실제로 들여다보니 우리 매출의 70%는 
삼삼엠투랑 리브에서 나오고 있었다.
거기엔 API도 없고, 자동화도 안 되고, 
재관이가 4개 창 열어놓고 머리로 판단하고 있었다.

그래서 방향을 바꿨다.
자동화가 아니라 "판단을 도와주는 도구"를 먼저 만들자.
70%는 사람이 결정하니까, 사람에게 좋은 근거를 주는 게 먼저다.

도시계획에서도 마찬가지였다. 
도시를 설계하는 건 시스템이 아니라 사람이다.
시스템은 사람이 더 나은 결정을 하도록 돕는 것.
```

**쓸 때 생각할 질문:**
- 이 작업에서 제일 놀라운 발견은?
- 내가 틀렸다고 느낀 순간은?
- 다음에 이 글을 다시 읽을 때 뭘 기억하고 싶은가?
- 도시공학과 이 기술은 어디서 만나는가?

---

### 3-2. 논문형

> 인과관계를 논리적으로 설명하고, 근거가 수렴되는 글

**목적**: 기술 결정의 논리적 정당화, 지식 축적
**대상**: 기술자, 투자자, 학계, 파트너
**톤**: 정제되고 구조적. 주장 → 근거 → 결론.

**구조 (필수):**

```
1. 배경
   - 이 문제가 왜 존재하는가
   - 기존에 어떻게 해결하고 있었는가

2. 문제 정의
   - 구체적으로 뭐가 안 되는가
   - 데이터로 보면 어떤 상태인가

3. 접근 방법
   - 어떤 선택지가 있었는가
   - 왜 이 방법을 선택했는가

4. 구현
   - 무엇을 만들었는가
   - 기술 구조는 어떠한가

5. 결과
   - 수치로 뭐가 바뀌었는가
   - 예상과 다른 점은

6. 시사점
   - 이 결과가 의미하는 바
   - 다음 연구/개발 방향
   - 한계와 제약

7. 참고 자료
   - 출처, 유사 사례, 외부 데이터
```

**이런 글이 나와야 한다:**

```
## 중단기 임대 채널의 가격 결정 구조 분석
### — 삼삼엠투 584건 거래 데이터 기반

### 1. 배경
글로벌 OTA 채널(Airbnb)에는 PriceLabs 같은 동적 가격 엔진이 존재한다.
그러나 국내 중단기 임대 채널(삼삼엠투, 리브애니웨어)은 
이 자동화 체계의 적용 범위 밖에 있다.

본 분석에서는 삼삼엠투 92개 숙소, 1,915건 계약 데이터를 바탕으로
국내 채널의 가격 구조와 체결 패턴을 정량적으로 분석한다.

### 2. 문제 정의
삼삼엠투의 게시 가격(주간 임대료)은 실질 비용의 60~75%만 반영한다.
관리비(주당 7~15만원), 청소비(1회 5~12만원), 보증금(33만원)이 
게시가에 포함되지 않아 가격 비교가 왜곡된다.

### 3. 분석 결과
584건 결제완료 거래를 기간별로 분류한 결과:
- 1주 체류: 실질가 대비 +11.5% (단기 프리미엄)
- 2주 체류: 실질가 대비 +6.7% (기준선)
- 3~4주: 실질가 대비 -8.4% (장기 할인)
- 5주+: 실질가 대비 -11.7% (대폭 할인)
...
```

**AI 보조 요청 사항:**
- 유사 사례 검색 (해외 중단기 임대 가격 연구)
- 통계적 유의미성 검증 (샘플 수, 편향)
- 출처 형식 (APA or 한국어 학술 형식)

---

### 3-3. 블로그

> 우리의 가치관과 기술을 대외에 알리는 글

**목적**: 브랜드 인지, 공감대 형성, 리드 생성
**대상**: 잠재 고객(숙소 오너), 업계 관계자, 일반인
**톤**: 친근하고 쉬운 말. 전문 용어는 풀어서. 핵심 메시지 반복.

**구조:**

```
1. 후킹 (공감 시작)
   "혹시 이런 경험 있으신가요?"
   일상적 상황에서 시작

2. 문제 제기
   이게 왜 문제인지 쉽게 설명
   "사실 이건 ~한 이유 때문입니다"

3. 우리의 접근
   어떻게 풀고 있는지
   너무 기술적이면 안 됨 — 비유, 예시 활용

4. 결과/인사이트
   "이렇게 해보니 이런 게 보였습니다"
   구체적 숫자 1~2개만

5. 마무리 메시지
   왜 이게 중요한지
   우리가 왜 이걸 하는지
   핵심 메시지 한 문장 반복
```

**이런 글이 나와야 한다:**

```
# 같은 방인데 왜 가격이 다를까?

서울 강동구에서 92개 숙소를 운영하고 있습니다.
에어비앤비에 올리면 1박에 11만원.
삼삼엠투에 올리면 1주에 35만원.
같은 방인데 채널마다 가격이 다릅니다.

왜 그럴까요?

에어비앤비는 1박 단위입니다. 청소비가 포함돼 있고요.
삼삼엠투는 1주 단위인데, 관리비가 별도예요. 청소비도 따로.
표면 가격만 보면 삼삼엠투가 싸 보이지만,
실제로 다 더하면 오히려 비슷하거나 더 비쌀 수도 있어요.

584건의 실제 거래를 분석해봤습니다.
1주 단위로 머무는 분은 평균 일 6.2만원을 내고,
8주 이상 장기 거주하시는 분은 일 4.9만원을 냅니다.

기간이 길수록 21% 저렴해지는 구조입니다.

우리는 이 데이터를 보면서 깨달았습니다.
가격을 자동으로 바꾸는 게 답이 아니라,
"이 방을 이 기간에 얼마에 팔아야 적정한지"를
운영자에게 알려주는 게 먼저라는 걸요.

도시에서 집값이 정해지는 방식과 비슷합니다.
같은 동네, 같은 평수여도 층, 방향, 시기, 수요에 따라 가격이 다르듯
숙소도 채널, 기간, 시즌, 옵션에 따라 적정 가격이 달라야 합니다.

우리는 그 "적정"을 데이터로 찾고 있습니다.
```

**핵심 메시지 반복 패턴:**
- 도입: "같은 방인데 왜 다를까?"
- 중간: "가격을 바꾸는 게 답이 아니라"
- 마무리: "적정을 데이터로 찾고 있습니다"

## 4단계: 검토/퇴고

> Founder(김진우) + CTO가 직접 다듬어서 컬럼으로 완성

이 단계는 AI가 아니라 **사람이** 한다.
- 에세이: 내 목소리가 살아있는지 확인. AI가 쓴 티 제거.
- 논문형: 논리 빈틈, 근거 부족 보완. 전문가 검수.
- 블로그: 읽는 사람 입장에서 통하는지 확인. 핵심 메시지가 남는지.

---

## 위키 운용 방식

### 저장 구조

하나의 위키 아티클에 대해 작업 기록부터 3종 초안까지 이어진다.

```
위키 아티클 #125: "매출 구조와 가격 전략"
│
├─ [작업 기록] 코드/설계/데이터 소스
├─ [에세이] 느낀 점, 배운 것
├─ [논문형] 구조적 분석
└─ [블로그] 대외 소통용
```

### 상태 매핑

| 위키 status | 아카이빙 단계 | 설명 |
|---|---|---|
| empty | - | 아직 아무것도 없음 |
| draft | 작업 기록 ~ 글 구성 | 소스 기록 + AI 초안 진행 중 |
| review | 초안 완성 ~ 검토/퇴고 | 3종 초안 작성됨, 사람 검토 대기 |
| published | 발행 | 위키 확정 |

### 주기

```
매 세션 종료 시:
  → 작업 기록 자동 저장 (Claude가 위키 POST)

주 1회 (금요일):
  → 주간 작업 기록 모아서 Gemini에 3종 초안 요청
  → Claude 검수 → 위키 저장

월 1회:
  → Founder + CTO 검토/퇴고
  → 블로그 1편 이상 외부 발행
```
