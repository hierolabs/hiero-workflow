# HIERO 선행지표 분석 — 가장 큰 퍼널의 첫 단계 찾기

> 작성일: 2026-05-11
> 작성: Claude (분석 + 가설)
> 검증 대상: 김진우 (Founder)

---

## TL;DR (3줄 요약)

1. **HIERO의 가장 큰 누수 퍼널은 "33m2 채팅 → 결제완료"**. 1,915건 시도 중 결제까지 가는 건 708건(37%) — **63%가 중간에 사라진다.**
2. **그 첫 단계는 "게스트 첫 메시지에 대한 호스트의 첫 응답"**. 이 지점이 모든 결과의 부모 노드인데, 현재 HIERO의 25개 진단 지표에는 측정되지 않는다.
3. **추천 매직 넘버**: 게스트 첫 메시지 후 **15분 이내 호스트 응답률**. 목표 70%. 이게 채워지면 결제 전환율과 가동률, 매출이 따라온다는 가설.

## 1. 출발점 — 왜 이 분석인가

영상의 핵심 메시지를 HIERO에 적용하면:

- **후행지표** = 매출, 가동률, ADR, 리뷰 점수
- **선행지표** = "결과의 부모가 되는 단일 행동"
- 페이스북의 "10일 내 친구 7명"처럼, HIERO만의 **매직 넘버**를 데이터에서 발견하는 것이 목표.

이 문서는 그 매직 넘버 후보를 찾는 작업이다.

## 2. "우리 고객"은 누구인가 — 두 갈래

HIERO는 B2B + B2B2C 비즈니스 구조다. 고객을 두 갈래로 분리해서 보면:

```
HIERO의 고객 구조

[Layer 1] HOST (임대인) — HIERO에게 운영 위탁
   ↓ HIERO가 운영
[Layer 2] GUEST (투숙객) — 숙소를 예약·이용
```

| | Layer 1 (Host) | Layer 2 (Guest) |
|---|---|---|
| 모델 | `outsourcing_lead` | `reservation` + `conversation` |
| 퍼널 시작 | DB 등록 (`new`) | 채팅 문의 (`chat_id`) |
| 퍼널 종료 | 계약 (`contracted`) | 결제완료 → 거주 → 정산 |
| 시도 볼륨 | 미상 (DB 미연결) | **1,915 / 6,364** (실측) |
| LTV | 매우 큼 (1 host = 12개월+ 매출) | 단발성 + 재방문 가능 |

**결론: 데이터가 풍부하고 누수가 가장 큰 쪽은 Layer 2 (Guest) 퍼널.** 분석은 여기에 집중. Layer 1은 별도 분석 권장.

## 3. HIERO의 5가지 퍼널 매핑

코드와 SYSTEM_MAP을 종합하면, HIERO에는 5개의 퍼널이 동시에 돌고 있다.

### 퍼널 A — 위탁영업 (Lead Acquisition)
```
new → contacted → replied → diagnosed → proposal_sent
→ contract_pending → contracted
```
- 시작: 외부 마케팅
- 끝: 신규 숙소 1개 추가
- 측정: `OutsourcingLead.Status`
- 누수 추정치: 미실측

### 퍼널 B — 숙소 온보딩 (Property Activation)
```
contracted → setting → filming → ota_registering
→ operation_ready → partially_active → fully_distributed → active
```
- 시작: 계약 완료
- 끝: 모든 플랫폼 active
- 측정: `Property.LifecycleStatus`
- 누수: SettingType별 3/5/14일 룰로 관리됨

### 퍼널 C — 게스트 예약 (Reservation) — Hostex
```
listing 노출 → 문의(메시지) → 예약 신청 → accepted
→ checked_in → checked_out → completed → review
```
- 시작: 채널 노출
- 끝: 리뷰 완료
- 측정: `Reservation.Status`, `Reservation.StayStatus`
- 볼륨: **6,364건** (실측)

### 퍼널 D — 게스트 계약 (33m2 mid-term)
```
chat 시작 (chat_id) → 계약대기 (contract_id) → 결제완료
→ 입주대기 → 거주중 → 계약종료 → 정산완료
```
- 시작: 게스트의 chat 문의
- 끝: 정산 완료
- 측정: 33m2 CSV
- 볼륨: **1,915건** (실측)

### 퍼널 E — 청소·이슈 (Operations)
```
checkout → cleaning_pending → assigned → dispatched
→ in_progress → completed (or → issue)
```
- 시작: 체크아웃
- 끝: 청소 완료
- 측정: `CleaningTask.Status`
- 운영 품질 → 리뷰 → 다시 예약 전환에 영향

## 4. 데이터로 본 가장 큰 퍼널

### 4-1. 33m2 데이터 (실측치, N=1,915)

```
시도 (chat 시작)        1,915  ████████████████████  100%
↓
결제완료                 708  ████████              37.0%   ← 63% 누수
↓
계약종료(정상 완주)        655  ███████               34.2%
↓
정산완료                 804  ████████              ← 일부는 취소-정산 포함
```

**누수의 56.7%는 "취소"** — 채팅을 시작했으나 결제까지 가지 못한 사례. 이게 HIERO 비즈니스의 가장 큰 단일 누수 지점.

### 4-2. 누수가 어디서 발생하는가? — 객실별 분포

전체 평균 취소율 59.6%인데, 객실별로 보면:

| 구분 | 객실 수 | 평균 취소율 | 시도 | 완료 |
|---|---|---|---|---|
| 취소율 최악 10 (강동 2룸 U94 등) | 10 | 83~100% | 113 | 9 |
| 취소율 평균대 (60%대) | ~80 | 55~70% | ~1100 | ~280 |
| 취소율 최저 10 (가산독산 L1 등) | 10 | 14~33% | 80 | 32 |

같은 운영팀, 비슷한 가격대인데 **객실별 취소율이 7배까지 벌어진다.** 이건 운영 방식(응답·정보 제공)의 차이가 결과를 좌우한다는 강한 신호다.

### 4-3. 가격 vs 취소율 — 가격이 비쌀수록 취소 ↑

| 주간 가격대 | 평균 취소율 | 완료율 |
|---|---|---|
| ~20만원 | 59.8% | 20.0% |
| 20-30만원 | 60.6% | 28.6% |
| 30-40만원 | 65.8% | 22.9% |
| 40-50만원 | 70.5% | 19.3% |
| 50만원+ | 76.0% | 21.2% |

가격이 오르면 취소율도 같이 오른다. 하지만 가장 결정적인 발견은 다음:

> **취소된 계약의 평균 금액 ₩1,135,617 > 완료된 계약의 평균 금액 ₩889,394**
>
> 즉, 비싼 거래가 더 잘 깨진다 = 큰 거래일수록 신뢰 형성이 부족하면 무너진다.

### 4-4. 게스트 유형 — 재시도자 패러독스

| 유형 | 시도 | 완료 | 완료율 |
|---|---|---|---|
| **1회 시도자** | 889 | 398 | **44.8%** |
| 여러 번 시도자 | 1,026 | 282 | 27.5% |

직관과 반대로, "이것저것 알아보는" 게스트는 덜 closing 된다.
**의미: 첫 시도에 명쾌하게 정보 제공하는 게 결정적.** 두 번 묻게 만들면 잃는다.

### 4-5. 결제 통과 후는 거의 안 깨진다

```
결제완료 → 계약종료: 655 / 708 = 92.5%
```

결제만 지나면 92.5%가 끝까지 간다. 즉, **돈을 받기 전의 시간이 진짜 funnel.** 그 안에서도 가장 첫 순간 = chat 첫 응답이 모든 것을 좌우한다.

현재 5엔진 × 5지표 = 25개 중에 **진짜 선행지표(매일 행동으로 바꿀 수 있는 양적 지표)는 거의 없다.**

선행 11개 중에서 **하루 단위로 측정·개선 가능한 지표는 5개 미만** (CSScore, CheckinScore, CleaningScore, AmenityScore, PriceFlexibility).

### 5-1. 빠진 진짜 선행지표 (가장 중요한 발견)

지금 25개 어디에도 없는, **결과의 부모 행동들**:

```
1. 첫 응답 시간 (Time to First Reply, TTFR) ← 1순위 후보
   - 게스트 첫 메시지 → 호스트 첫 응답 사이의 분 단위 시간
   - conversations + messages 테이블에 이미 sent_at 있음
   - 즉시 계산 가능

2. 첫 시간 메시지 왕복 수 (First-Hour Round Trips)
   - 첫 1시간 안에 몇 번 주고받았는가
   - >3회면 진지한 문의로 분류 가능

3. 핵심 질문 4종 응답 완성도
   - 가격, 공실, 위치, 옵션 — 첫 응답에 몇 개 포함되었는가
   - 활동 로그에 "parking 문의 이슈 생성" 패턴이 이미 보임

4. 체크인 안내 발송 시점 (Pre-checkin Message Lead Time)
   - 체크인 X시간 전 발송 → 입실 트러블 감소

5. 리뷰 답글 작성률
   - 리뷰가 달린 후 호스트가 답글 단 비율
   - 다음 게스트 검색 결과에 직접 영향
```

이 다섯이 빠져 있다는 것이 가장 큰 진단 결과.

## 6. 첫 단계 후보 — 매직 넘버 가설 5개

### 가설 H1 ⭐ (최우선) — 첫 응답 시간 (TTFR)

```
정의: 게스트의 첫 메시지가 들어온 시점 → 호스트의 첫 응답 시점
측정: messages.sent_at(sender_type='guest', 첫 행)
     vs messages.sent_at(sender_type='host', 첫 행)
가설: 15분 이내 응답 시 결제 전환율 X배

목표 매직 넘버: TTFR ≤ 15분 → 결제 전환율 70%+
```

**왜 1순위인가:**
- 모든 다른 결과의 부모 노드 (응답 없으면 이후 단계 진행 불가)
- 1,915건 100%가 chat에서 시작 (이미 확인)
- 1회 시도자가 더 잘 closing 되는 데이터와 일치 (즉답이 결정적)
- 이미 데이터로 측정 가능, 즉시 시작

### 가설 H2 — 핵심 4정보 답변율

```
정의: 첫 응답에 가격·공실·위치·옵션 중 몇 개를 명시했는가
측정: 첫 응답 메시지 텍스트에 4가지 키워드 패턴 매칭
가설: 4/4 포함 시 결제 전환율 +20%p

목표 매직 넘버: 4/4 명시 응답률 60%+
```

**근거:** "parking 문의 이슈 생성"이 activity_log에 반복적으로 떠 있음. 게스트의 같은 질문이 반복된다는 것은 첫 응답에서 답을 못 주고 있다는 신호.

### 가설 H3 — 첫 24시간 내 결제 유도

```
정의: chat 시작 후 24시간 이내 결제완료된 비율
측정: chat_id 시작 시각 vs payment_status=결제완료 도달 시각
가설: 24시간 초과 시 취소율 급증

목표 매직 넘버: 24시간 내 결제율 50%+
```

**근거:** 결제만 통과하면 92.5%가 끝까지 간다. 결제까지 끌고 가는 시간이 짧을수록 좋다.

### 가설 H4 — 체크인 X시간 전 안내 발송률

```
정의: 체크인 24시간 전에 안내 메시지를 발송한 예약의 비율
측정: messages(체크인 안내 텍스트 매칭) vs reservation.check_in_date
가설: 발송 시 리뷰 점수 +0.5, 클레임 -30%

목표 매직 넘버: 24시간 전 안내 발송률 90%+
```

**근거:** 5엔진의 CheckinScore가 있지만 입력값(=행동의 양)이 정의돼 있지 않음. 이걸 "체크인 안내 발송률"로 정의하면 측정·운영·개선 가능.

### 가설 H5 — 리뷰 답글 작성률

```
정의: 게스트 리뷰가 달린 후 7일 내 호스트가 답글을 단 비율
측정: review.guest_review_at vs review.host_reply
가설: 답글 있는 listing의 다음 달 예약율 +15%

목표 매직 넘버: 7일 내 답글 작성률 80%+
```

**근거:** ReviewScore는 후행이지만, "답글 작성"은 호스트의 행동(선행). 직접 통제 가능.

## 7. 추천 — 지금 당장 측정 시작할 1개

**H1 (TTFR — Time to First Reply) 을 첫 단계로 추천**.

### 왜 H1인가

| 평가 축 | H1 | H2 | H3 | H4 | H5 |
|---|---|---|---|---|---|
| 이미 데이터 있음 | ✅ | ✅ | ✅ | △ | ✅ |
| 1주일 안에 계산 가능 | ✅ | △ | ✅ | △ | ✅ |
| 결과의 부모 위치 | ⭐⭐⭐ | ⭐⭐ | ⭐⭐ | ⭐ | ⭐ |
| 단일 숫자로 추적 가능 | ✅ | △ | ✅ | ✅ | ✅ |
| 운영팀이 즉시 행동 가능 | ✅ | ✅ | △ | ✅ | ✅ |
| 다른 후행지표 연쇄 개선 | ⭐⭐⭐ | ⭐⭐ | ⭐⭐ | ⭐⭐ | ⭐ |

**H1은 다른 모든 가설의 부모 노드**다. TTFR이 빠르면:
- H2 (4정보 답변)을 빠르게 깔 수 있고
- H3 (24시간 결제)의 출발선이 앞당겨지고
- H4·H5 (체크인·리뷰 답글)와 같은 "응답 습관"이 같은 호스트팀에서 이뤄지므로 함께 개선된다.

### 실행 단계 (4주 스프린트)

```
Week 1: 측정
  - 백엔드에 TTFR 계산 endpoint 추가
  - 객실별·호스트별 TTFR 분포 시각화
  - 베이스라인 확정 (현재 중앙값, P90)

Week 2: 라벨링
  - TTFR 15분 이내 vs 초과 그룹으로 분리
  - 각 그룹의 결제 전환율 + 취소율 비교
  - 가설 검증: 차이가 유의미한가?

Week 3: 실험
  - TTFR 15분 룰을 운영팀(예린/재관)에 적용
  - 알림: 채팅 미응답 10분 시 푸시
  - 5엔진 진단의 26번째 지표로 추가 (CSScore 분해)

Week 4: 정착
  - 매일 대시보드에 TTFR 표시 (가동률 옆)
  - 호스트별 TTFR 랭킹
  - 위키 8탭 파이프라인의 "작업 기록" 탭에 자동 누적
```

## 8. 25개 지표 → 26번째 지표 추가 제안

현재 5엔진 25지표를 그대로 두고, **4번 엔진(Value Delivery)에 1개를 추가**한다.

```go
// PropertyBusinessDiagnosis 4번 엔진 — Value Delivery (현재 5개 → 6개)
CleaningScore int    // 청소 품질
CheckinScore  int    // 체크인 편의성
CSScore       int    // CS 응답 (점수)
AmenityScore  int    // 비품 관리
ClaimRate     int    // 클레임 발생률
FirstReplyMin int    // ⭐ 신규: 첫 응답 분 (중앙값, 낮을수록 좋음)
```

**왜 4번 엔진인가:** 5엔진은 가치 창출 → 마케팅 → 판매 → **전달** → 재무 순. "첫 응답"은 약속한 가치를 게스트에게 전달하는 첫 접점. Value Delivery에 속한다.

**왜 점수가 아닌 "분"인가:** 0~100 점수는 추상적이고 비교 불가. "분 단위 중앙값"은 객관적이고 운영팀이 직접 줄일 수 있다.
